# 如何通俗易懂的给小白解释零知识证明？ - 知乎
如果有人问你”你满18了吗”，你通常只有两个选择：要么掏出身份证，把姓名、住址、身份证号全都暴露给对方；要么干脆拒绝回答。

但有没有第三种可能——你不透露生日、不出示身份证、甚至不告诉对方你的名字，却让对方100%确信你已满18岁？

这不是魔术，是数学。准确地说，是一种叫做[零知识证明](https://zhida.zhihu.com/search?content_id=770936485&content_type=Answer&match_order=1&q=%E9%9B%B6%E7%9F%A5%E8%AF%86%E8%AF%81%E6%98%8E&zhida_source=entity)（Zero-Knowledge Proof, [ZKP](https://zhida.zhihu.com/search?content_id=770936485&content_type=Answer&match_order=1&q=ZKP&zhida_source=entity)）的密码学技术。它在2025-2026年间从学术论文走向了现实部署：Google在Wallet里上线了ZKP年龄验证，丹麦即将推出全球首个国家级ZKP系统，欧盟计划2026年底在27个成员国强制部署。

年龄验证这件事，正在被密码学彻底重写。

* * *

一、年龄验证的”不可能三角”
--------------

先看看现在的年龄验证方案有多烂。

**身份证上传。**  你把身份证正反面拍照上传给一个网站。准确吗？准确。但你的姓名、住址、身份证号、出生日期全部暴露了。2024年澳大利亚一家年龄验证服务商的数据库被攻破，数十万用户的身份证照片泄露。上传身份证做年龄验证，本质上是用核弹打蚊子——你只想证明”我满18了”，却把全部个人信息都交了出去。

**面部年龄估计。**  AI扫描你的脸，估算你的年龄。不需要上传证件，用户体验不错。但准确率有上限——16岁长得老成的人可以轻松通过，40岁保养好的人可能被拦住。更关键的是，你的面部生物特征数据被采集了。这些数据一旦泄露，你不能”换一张脸”。

**信用卡验证。**  简单粗暴，有信用卡就当你是成年人。绕过方式太多——借用父母的卡、使用预付卡——准确性极差。

**自我声明。**  “请确认您已满18岁，点击确认。”这纯粹是法律合规的形式主义，和年龄验证没有任何关系。一个12岁的孩子点击”确认”不需要任何技术能力。

**手机号验证。**  用手机号注册时绑定实名信息。问题是手机号本身就是一个强关联标识符——运营商知道你是谁，网站知道你的手机号，两边一对就能还原完整身份。而且未成年人用父母的手机号注册毫无障碍。

**第三方年龄验证中间商。**  听起来专业，但本质上是引入了一个新的信任方。中间商同时知道你的身份信息和你要访问的网站——它比任何一方都知道得更多。一旦中间商的数据库被攻破或者中间商本身出售数据，所有用户的隐私全部归零。

把这些方案放在一起看，你会发现一个清晰的不可能三角：

|  | 准确性 | 隐私 | 用户体验 | 绕过难度 |
| --- | --- | --- | --- | --- |
| 身份证上传 | 高 | 极差 | 差 | 中 |
| 面部年龄估计 | 中-高 | 中 | 好 | 中 |
| 信用卡验证 | 低 | 中 | 好 | 低 |
| 自我声明 | 无 | 好 | 好 | 无 |
| ZKP + DID | 高 | 极好 | 中 | 高 |

准确性、隐私、用户体验——传统方案最多只能兼顾两个。想要准确就得牺牲隐私，想要隐私就牺牲准确性，想要好体验就两个都得放弃。

零知识证明的目标，是用密码学打破这个三角。

二、零知识证明到底是什么？
-------------

零知识证明的核心思想用一句话就能说清楚：**我可以向你证明一个陈述是真的，而不透露除此之外的任何信息。** 

具体到年龄验证：我可以向你证明”我已满18岁”，而不告诉你我到底几岁、我叫什么名字、我的生日是哪天。你得到的信息有且只有一个比特：yes或no。

这听起来违反直觉，但它有严格的数学基础。零知识证明需要满足三个密码学条件：

1.  **完备性（Completeness）**：如果你真的满18岁，你一定能通过验证。不会误杀。
2.  **可靠性（Soundness）**：如果你没满18岁，你（几乎）不可能伪造一个能通过验证的证明。不会漏网。
3.  **零知识性（Zero-Knowledge）**：验证者除了知道”这个人满18岁”之外，学不到任何其他信息。没有附带伤害。

让这三个条件同时成立的机制是一个”双盲”架构，涉及三个角色：

**第一步：信任锚签发凭证。**  政府（作为信任锚）签发一个数字凭证，内容类似”持有者的出生年份是1998年”。这个凭证用政府的密钥签名，不可伪造。

**第二步：持有者生成证明。**  你的手机钱包拿到这个凭证后，在本地——注意是在你自己的设备上——生成一个零知识证明。这个证明的内容是”根据政府签发的凭证，我已满18岁”。。

**第三步：验证者验证证明。**  网站收到这个证明后，可以在毫秒级验证其有效性。验证过程只需要政府的公钥——不需要看到原始凭证，不需要知道你是谁。

关键在于三方的信息隔离：**政府不知道你访问了哪个网站**（因为你是离线生成证明的），**网站不知道你是谁**（因为它只看到一个数学证明）。整个流程中，没有任何一方掌握完整信息。

用代码来理解可能更直观。以下是一个用Circom语言编写的简化版年龄验证电路：

```text
template AgeVerification() {
    signal input birthYear;     // 私有输入（不可见）
    signal input currentYear;   // 公有输入
    signal input minAge;        // 公有输入
    signal output isOldEnough;  // 输出

    signal age;
    age <== currentYear - birthYear;
    isOldEnough <== age >= minAge;
}
```

`birthYear`是私有输入——验证者看不到。`currentYear`和`minAge`是公有输入——双方都知道。电路计算出年龄差，判断是否满足最低年龄要求，输出一个布尔值。零知识证明系统保证：验证者可以确认这个电路被正确执行了，但无法反推出`birthYear`

数学上是怎么做到的？简单说，证明者把计算过程转化为一组多项式方程，然后用椭圆曲线密码学生成一个简短的”承诺”（commitment），验证者可以检查这个承诺是否与正确的计算一致，但无法从中提取原始输入。具体的数学涉及双线性配对、Schwartz-Zippel引理等——细节超出本文范围，但核心直觉就是：**验证一个计算是否正确，比执行这个计算本身更容易，而且不需要看到输入数据。**

三、三条技术路线：zkSNARKs、zkSTARKs、Bulletproofs
---------------------------------------

零知识证明不是一种技术，而是一个技术家族。目前主流的三条路线各有取舍。

### zkSNARKs

全称”Zero-Knowledge Succinct Non-Interactive Argument of Knowledge”。”Succinct”是关键词——证明非常小，只有几KB。

优势很突出：证明体积最小，验证速度最快。在移动端生成一个年龄证明只需要220-285毫秒。对于需要在手机上实时生成证明的年龄验证场景，这个速度至关重要——用户点一下按钮，眨眼之间完成。

代价也明确：zkSNARKs需要一个”可信设置”（trusted setup）阶段。在系统启动之前，需要一组参与者生成公共参数，然后销毁生成这些参数时使用的随机数。如果这些随机数没有被真正销毁——比如某个参与者保留了备份——那么他就可以伪造证明。Zcash在2016年用了一个精心设计的多方计算仪式来完成可信设置，但这个问题始终是一个信任假设。

另一个问题：zkSNARKs使用的椭圆曲线密码学不抗量子计算攻击。量子计算机目前还不够强，但10-15年后就不好说了。对于需要长期运行的国家级身份系统，这是一个必须考虑的因素。

### zkSTARKs

全称”Zero-Knowledge Scalable Transparent Argument of Knowledge”。”Transparent”是关键词——不需要可信设置。

zkSTARKs用哈希函数代替椭圆曲线，这意味着两件事：第一，不需要可信设置仪式，系统的安全性不依赖于任何人的诚实行为；第二，哈希函数被认为是抗量子攻击的，这让zkSTARKs成为面向未来的选择。

代价是证明体积大得多——通常是数百KB级别，比zkSNARKs大两个数量级。对于链上验证或带宽受限的场景，这是一个实际问题。对于年龄验证这种只需要传一次证明的场景，影响相对可控。

### Bulletproofs

不需要可信设置，证明体积比zkSTARKs小得多，网络影响最小。看起来很完美？

但速度最慢。Bulletproofs的验证时间随电路规模线性增长，而zkSNARKs和zkSTARKs的验证时间是常数或接近常数。对于简单的年龄比较，Bulletproofs够用；对于更复杂的场景（比如同时验证年龄和居住地），性能会成为瓶颈。

**实际选择取决于场景。**  Google Wallet选了zkSNARKs——因为手机端需要毫秒级生成。国家级系统可能倾向zkSTARKs——因为不依赖可信设置、抗量子攻击。范围证明（仅证明年龄在某个区间内）可以用Bulletproofs——因为不需要完整的电路计算。

不存在”最好的ZKP方案”，只有”最适合特定场景的方案”。三者的核心取舍总结如下：

|  | 证明大小 | 生成速度 | 需要可信设置 | 抗量子 |
| --- | --- | --- | --- | --- |
| zkSNARKs | 最小（数KB） | 最快（220-285ms） | 是 | 否 |
| zkSTARKs | 最大（数百KB） | 中等 | 否 | 是 |
| Bulletproofs | 中等 | 最慢 | 否 | 部分 |

对于年龄验证这个特定场景——证明逻辑简单、移动端生成、一次性传输——zkSNARKs目前占优。但如果欧盟要建一个运行几十年的国家级身份系统，zkSTARKs的”无可信设置+抗量子”组合可能更合适。技术选型永远是工程权衡，不是学术比赛。

四、不再是论文里的东西
-----------

说到这里，你可能觉得这些都是密码学家的智力游戏。两年前确实是。但2024-2026年间，ZKP年龄验证的落地速度超出了大多数人的预期。

### Google Wallet

2025年5月，Google在Google Wallet中正式部署了ZKP年龄验证。这不是实验项目，是面向普通用户的生产环境。首个合作伙伴是约会应用Bumble——用户可以在不透露具体年龄的情况下，向Bumble证明自己已满18岁。

2025年7月，Google更进一步，将其ZKP库开源。这意味着任何开发者都可以使用Google验证过的ZKP实现，而不需要自己从零搭建密码学基础设施。

一家市值2万亿美元的公司在其核心产品中部署ZKP，并且选择开源——这不是学术探索，这是基础设施建设。Google的意图很明确：让ZKP年龄验证变成像HTTPS一样的行业标配，而不是少数密码学团队才能实现的高端技术。开源降低了所有开发者的接入门槛，这对整个生态的加速意义不亚于技术本身。

### 丹麦[AltID](https://zhida.zhihu.com/search?content_id=770936485&content_type=Answer&match_order=1&q=AltID&zhida_source=entity)

丹麦正在建设全球首个国家级ZKP年龄验证系统，品牌名AltID，计划2026年春季上线。

这个系统线上线下通用。在便利店买酒？扫一下手机，店员看到”已满18”——看不到你的名字、地址或出生日期。在网站注册？同样的流程，网站只收到一个数学证明。

丹麦选择第一个吃螃蟹，一个重要原因是该国已有成熟的数字身份基础设施（MitID）。ZKP不能凭空运作——它需要一个可信的数字身份作为起点。丹麦恰好具备这个前提条件。

### EU [EUDI钱包](https://zhida.zhihu.com/search?content_id=770936485&content_type=Answer&match_order=1&q=EUDI%E9%92%B1%E5%8C%85&zhida_source=entity)

欧盟的EUDI（European Digital Identity）钱包计划更具野心。2026年底前，所有27个成员国必须部署数字身份钱包，其中包含ZKP实验特性。

这不是可选项。这是欧盟法规要求的强制部署。4.5亿人的市场，统一的数字身份基础设施，原生支持零知识证明——如果这个计划按时落地，它将成为全球最大规模的ZKP部署。

这里有一个有趣的监管逻辑：欧盟不是在说”你可以用ZKP”，而是在说”你必须部署数字身份钱包，而这个钱包必须支持选择性披露”。选择性披露的技术实现，最自然的选择就是ZKP。法规没有直接要求ZKP，但它要求的功能特性几乎只有ZKP能满足。这是一种”通过功能需求间接推动技术采用”的监管策略。

### [World ID](https://zhida.zhihu.com/search?content_id=770936485&content_type=Answer&match_order=1&q=World+ID&zhida_source=entity)

Tools for Humanity的World ID基于[Semaphore协议](https://zhida.zhihu.com/search?content_id=770936485&content_type=Answer&match_order=1&q=Semaphore%E5%8D%8F%E8%AE%AE&zhida_source=entity)实现匿名身份验证，已经拥有超过500万用户。它与Tinder合作，用于年龄验证和真人证明。

Semaphore协议值得单独说一下。它由Privacy & Scaling Explorations（PSE）团队构建，实现的是匿名群组成员证明。具体流程是：政府创建一个”18岁以上”群组，满足年龄要求的用户加入这个群组，需要验证时用户生成一个证明——”我是这个群组的成员”——但不暴露自己是群组中的哪一个人。

Semaphore还内置了一个叫nullifier的防重复信号机制。它确保同一个用户不能在同一个场景中重复使用证明（防止一个人帮多个人验证），同时又不会暴露用户身份。这是一个非常精巧的密码学设计。

### 其他参与者

**Polygon ID（现更名为Privado ID）** 已经在生产环境运行，基于Iden3协议和Circom工具链。它的定位是区块链原生的去中心化身份基础设施。

**Concordium** 是一条L1区块链，直接在协议层内置了ZKP身份层——不是作为应用层的附加功能，而是区块链本身的基础能力。

**euCONSENT的AgeAware项目** 在2025年4月完成了概念验证，专门面向欧洲市场的年龄验证需求。

**ISO/IEC 27566-1:2025** 在2025年12月发布——这是首个全球年龄验证标准，并且免费提供。一个ISO标准免费公开，本身就说明了行业推动这项技术普及的决心。

从Google这样的巨头到丹麦这样的小国，从区块链项目到国际标准化组织，ZKP年龄验证的推动者覆盖了技术栈的每一层。这种多线并进的格局，通常意味着技术已经越过了”是否可行”的争论阶段，进入了”如何落地”的执行阶段。

五、法国的”双重匿名”——为什么这才是正确的架构
------------------------

2025年4月，法国媒体监管机构Arcom发布了一项强制要求：所有年龄验证系统必须满足”双盲架构”（double-blind architecture）。

这两个”盲”分别是：

**第一盲：网站不知道用户身份。**  网站只能知道”这个人已满18岁”，不能获取任何可识别身份的信息。没有姓名，没有生日，没有身份证号。

**第二盲：身份提供者不知道用户访问了什么。**  签发年龄凭证的机构不能知道用户拿着这个凭证去了哪个网站、在什么时间访问的。

两个条件同时满足，意味着没有任何一方能够把”某个人”和”某个网站的某次访问”关联起来。

这个要求看似简单，但它实际上排除了绝大多数传统年龄验证方案。

身份证上传？网站看到了你的全部信息，第一盲不满足。中心化的年龄验证中间商？中间商同时知道你是谁和你去了哪个网站，两个盲都不满足。OAuth式的联合登录？身份提供者可以追踪你的登录行为，第二盲不满足。

能够天然满足双盲要求的技术路线，本质上只有两种：零知识证明，或者具有类似隐私属性的密码学协议。

法国的双盲标准不是技术建议，是法律要求。它把”隐私保护”从一个可选的道德目标，变成了一个可执行的合规门槛。任何想在法国市场运营年龄敏感服务的公司，要么用ZKP，要么用功能等价的方案——没有第三条路。

这个标准如果被其他欧盟国家效仿——考虑到EUDI钱包的推进节奏，这几乎是必然的——那么双盲架构将成为欧洲年龄验证的事实标准。而ZKP是目前实现双盲架构最成熟的技术选项。

值得注意的是，法国的双盲要求不仅仅是为了年龄验证。它建立了一个更广泛的原则：任何数字身份验证都不应该让任何单一方获得完整的用户画像。这个原则如果被推广到其他身份属性——国籍、职业资格、信用评分——那么ZKP将不仅仅是年龄验证的工具，而是整个数字身份体系的隐私基础层。

六、[EFF](https://zhida.zhihu.com/search?content_id=770936485&content_type=Answer&match_order=1&q=EFF&zhida_source=entity)的冷水：ZKP不是万能的
------------------------------------------------------------------------------------------------------------------------------------

到目前为止，ZKP听起来像是隐私问题的终极解药。但电子前哨基金会（EFF）在2025年7月泼了一盆必要的冷水。

EFF的原话：”零知识证明本身不是保护用户隐私的数字身份解决方案。”

这不是说ZKP没用，而是说单靠ZKP远远不够。问题出在元数据上。

### 元数据泄露

ZKP保护的是证明的内容——验证者确实看不到你的生日和姓名。但ZKP不保护证明的元数据。

**凭证签发方身份。**  你的年龄凭证是加州DMV签发的还是纽约DMV签发的？这个信息通常是公开的（验证者需要知道该信任谁的签名）。但”加州DMV签发”就意味着你住在加州。一个比特的信息就把4000万人的范围缩小到了一个州。

**时间戳和网站URL。**  验证发生的时间和目标网站是可观察的。如果你的ISP或网络中间人在监控流量，他们知道”某个IP在某个时间访问了某个网站并提交了年龄验证”。ZKP保护了证明的内容，但没有保护传输证明这个行为本身。

**IP地址和浏览器指纹。**  这是最基本也是最常被忽视的问题。你用ZKP证明了年龄，但你的IP地址暴露了你的大致位置，你的浏览器指纹（屏幕分辨率、安装的字体、浏览器版本的组合）可以在90%以上的概率下唯一标识你。

### 时序关联攻击

这是一个更微妙的威胁。假设你在A网站证明了”我满18岁”，在B网站证明了”我满21岁”，在C网站证明了”我未满25岁”。单看每一次验证，都没有泄露生日。但把三次验证的结果组合起来——满18、满21、未满25——你的出生年份就被缩小到了一个三年的窗口。

如果攻击者能够关联这三次验证属于同一个人（通过IP地址、浏览器指纹或其他标识符），那么”零知识”在实践中就变成了”少量知识”。多次独立的零知识证明叠加在一起，可以逐步缩小精确出生日期的范围。

### ZKP需要搭配使用

EFF的结论不是”不要用ZKP”，而是ZKP必须与其他隐私保护技术组合使用：

*   **Tor或VPN**：隐藏IP地址，防止网络层的身份关联
*   **反浏览器指纹技术**：防止通过浏览器特征识别用户
*   **凭证轮换机制**：定期更换凭证，增加时序关联的难度
*   **统一阈值设计**：避免不同网站使用不同的年龄阈值（都用18岁），减少通过组合推断的信息量

ZKP解决的是证明层的隐私问题。但一个完整的隐私保护系统还需要解决网络层、设备层、行为层的隐私问题。忽略这些层，ZKP就像是给房子装了一扇防弹门，但窗户全开着。

* * *

七、还有哪些坑
-------

除了EFF指出的元数据问题，ZKP年龄验证还面临几个结构性挑战。

**协议实现的脆弱性。**  ZKP的数学理论是严密的，但从数学到代码的翻译过程充满了出错的可能。电路约束写漏了一个条件、随机数生成器不够随机、椭圆曲线参数选错了——任何一个实现层面的bug都可能让整个系统的安全保证瞬间归零。目前很多ZKP协议的实现还没有经过严格的形式化验证，”理论上安全”和”实践中安全”之间的鸿沟依然存在。

**对政府数字身份的依赖。**  ZKP年龄验证的前提是：你有一个政府签发的数字身份凭证。丹麦有MitID，欧盟在推EUDI——发达国家的基础设施正在就位。但全球仍有超过10亿人没有任何形式的官方身份证明。对于发展中国家缺乏身份基础设施的用户来说，ZKP年龄验证是一个他们根本无法使用的系统。技术再先进，如果它的前提条件把一部分人排除在外，那它就不是一个普适的解决方案。

**用户体验的代差。**  虽然zkSNARKs的证明生成只需要200多毫秒，但整个流程——打开钱包应用、选择凭证、授权生成证明、提交给网站——比”点击确认我已满18岁”复杂得多。对于普通用户来说，任何增加摩擦的验证方式都会面临抵触。Google Wallet的集成方式在降低这个摩擦，但距离”无感验证”还有明显的差距。

**互操作性的碎片化。**  Google Wallet用一套协议，World ID用Semaphore，Polygon ID用Iden3，各家的ZKP实现互不兼容。一个用户如果在不同平台有不同的数字身份，需要维护多个钱包、多套凭证。ISO/IEC 27566-1:2025标准的发布是朝着统一方向迈出的一步，但从标准发布到行业统一采用，历史经验告诉我们这个过程通常需要3-5年。

**法律框架的滞后。**  技术已经能做到的事情，法律还没准备好。如果一个ZKP年龄验证系统出了问题——比如一个未成年人通过了验证——责任在政府（签发凭证的一方）、用户（生成证明的一方）、还是网站（验证证明的一方）？三方信息隔离在技术上是优势，但在法律追责上变成了障碍。这个问题在丹麦和欧盟的立法讨论中还在争论。

* * *

结尾
--

把所有信息拉到一起看。

年龄验证存在一个准确性-隐私-体验的不可能三角。传统方案在这个三角里来回取舍，每一种都要牺牲点什么。ZKP的数学性质让它成为目前唯一能在三个维度上同时得高分的技术路线。

Google、丹麦、欧盟、World ID的部署证明这不再是理论推演。ISO标准已经发布，法国的双盲要求已经写入法规。2026年将是ZKP年龄验证从”先行者实验”变成”行业基础设施”的转折年。

但EFF的警告同样重要。ZKP保护了证明的内容，却不保护证明的过程。元数据泄露、时序关联攻击、协议实现的脆弱性——这些问题不会因为底层密码学的优雅而自动消失。ZKP是年龄验证的正确方向，但不是银弹。它需要与VPN、反指纹技术、凭证轮换等手段组合使用，需要严格的形式化验证，需要政府投资数字身份基础设施。

最终的判断：ZKP是年龄验证领域几十年来最重要的技术突破。它第一次在数学层面证明了”验证身份”和”暴露身份”可以被彻底解耦。但从数学突破到真正的隐私保护，中间还隔着工程实现、基础设施建设和监管协调的漫长路程。

好消息是，这条路已经有人在走了——而且走得比预期快得多。
